Бизнес-цели MVP

    Перевести запись в онлайн — клиенты сами выбирают и бронируют слоты без участия администратора.

    Исключить двойные бронирования — задача бэкенда, но приложение должно корректно отображать статус мест и обрабатывать отказы.

    Снизить нагрузку на Дениса — вся рутина (просмотр расписания, запись, отмена, напоминания) автоматизируется.

    Собрать обратную связь — оценки маршалам для контроля качества.

    Быть готовым к сезону — релиз через 2 месяца с базовым, но стабильным функционалом.

1. Функциональные требования
1.1. Просмотр расписания (каталог слотов)

    F-001. Отображение списка заездов. Приложение показывает слоты (заезды) на ближайшие 7 дней по умолчанию (горизонт планирования). Если на выбранный период нет слотов — отображается пустое состояние с текстом «Пока нет доступных заездов».

    F-002. Фильтр по дате. Пользователь может выбрать конкретную дату или расширить период для просмотра.

    F-003. Карточка заезда (слота). В карточке отображается:

        Дата и время старта.

        Конфигурация трассы (короткая/длинная) с пометкой для новичков или опытных.

        Имя маршала-инструктора, который ведет заезд.

        Количество свободных картов (из 14), с учетом ограничения в 8 человек для новичковых заездов (этот лимит приходит с бэкенда).

        Доступность прокатной экипировки (шлем, подшлемник) — количество свободных комплектов.

    F-004. Статусы слота. Слот может быть:

        Доступен для бронирования.

        Заполнен / мест нет.

        Отменен центром (с указанием причины, например, «погодные условия»). Повторная запись на такой слот запрещена.

1.2. Бронирование заезда

    F-005. Выбор количества участников. Клиент выбирает количество мест (с учетом доступных картов).

    F-006. Выбор экипировки. На этапе бронирования клиент отмечает: «Я со своей экипировкой» или «Нужен прокат». При выборе проката система проверяет, достаточно ли свободных комплектов на этот слот (данные приходят из бэкенда).

    F-007. Подтверждение бронирования. При нажатии «Записаться» приложение отправляет запрос на бэкенд. Бэкенд атомарно проверяет свободные места.

        Успех: Приложение показывает подтверждение.

        Отказ (мест нет): Приложение показывает понятное сообщение («К сожалению, свободные карты закончились. Пожалуйста, выберите другой заезд»).

    F-008. Обработка двойных броней. Приложение не пытается "зарезервировать" слот локально — оно полагается на ответ бэкенда и корректно обрабатывает ситуацию, если места внезапно закончились.

1.3. Управление бронированием (отмена)

    F-009. Отмена клиентом. Клиент может отменить свою бронь через приложение.

    F-010. Отмена центром/по погоде. Если заезд отменен центром, бронь не удаляется из истории, а получает статус «Отменен центром» с указанием причины. Клиенту приходит push-уведомление об этом.

    F-011. Ограничение на позднюю отмену. (Опционально для MVP, если бэкенд это поддерживает). При попытке отмены менее чем за N минут до старта показывается предупреждение: «Отмена менее чем за 10 минут невозможна. Если вы не приедете, карт будет простаивать» (или аналогичное сообщение в зависимости от политики центра). Сама возможность отмены регулируется бэкендом.

1.4. Оценка маршала (после заезда)

    F-012. Оценка и отзыв. После даты и времени заезда клиенту в приложении доступна карточка заезда, где он может:

        Поставить оценку маршалу (например, от 1 до 5 звезд).

        Оставить короткий текстовый комментарий (опционально).

1.5. Уведомления

    F-013. Push-уведомления. Приложение отправляет (через интеграцию с бэкендом) клиенту:

        Напоминание о предстоящем заезде (например, за 2 часа).

        Уведомление об отмене заезда центром (по погоде или другой причине).

1.6. Профиль и история

    F-014. Личный кабинет. Клиент видит:

        Свои предстоящие бронирования (с датой, временем, трассой).

        Историю прошедших заездов.

    F-015. Отметка постоянных клиентов. Для MVP — просто отображение бейджа «Постоянный клиент» в профиле и, возможно, небольшая скидка (если эта логика есть в бэкенде). Это пока скорее визуальное "поглаживание".

2. Нефункциональные требования

    N-001. Платформа. Клиентское мобильное приложение (кросс-платформенное, например, на Flutter или React Native — чтобы закрыть и iOS, и Android).

    N-002. UX/UI. Приоритет — мобильный телефон. Интерфейс должен быть понятным, крупные кнопки (клиенты будут пользоваться у трассы, на солнце). Минимум текста, максимум визуальных подсказок (цветовые индикаторы занятости).

    N-003. Производительность. Приложение должно работать быстро даже при плохом мобильном интернете (кеширование списка слотов на несколько минут).

    N-004. Надежность. Приложение устойчиво к сбоям сети. Если нет интернета — показывать кешированные данные и сообщать, что требуется соединение для бронирования.

    N-005. Безопасность. Взаимодействие с API только по HTTPS. Авторизация клиентов (через номер телефона или соцсеть — базовая, без сложных схем).

    N-006. API-контракт. Все взаимодействия с бэкендом строго по утвержденному API-контракту (Swagger/OpenAPI). Мы не внедряемся в логику бэкенда, а только потребляем его данные и отправляем команды.

3. Что НЕ входит в MVP (отложено)

    Онлайн-оплата. Принимаем только бронирование. Оплата позже (наличные/перевод на карту).

    Интерфейсы маршала и администратора. Только клиент. Управление расписанием и маршалами — через существующую инфраструктуру.

    Сложная система лояльности. Только базовое отображение статуса.

    Интеграция с погодными сервисами. Отмена по погоде инициируется вручную администратором в бэкенде, а приложение только показывает статус и отправляет уведомление.

    Глубокое редактирование профиля. Только база.

    Исторические данные. Считаем, что база данных (слоты, маршалы) стартует с чистого листа или с того момента, как мы договорились о контракте. Миграция старых записей (из Telegram и доски) не делается.

4. Критерии приемки MVP (Done Done)

    Клиент видит список доступных заездов на неделю вперед.

    Клиент может записаться на заезд, выбрав количество людей и опцию проката экипировки.

    Система предотвращает запись, если свободных картов (или экипировки) нет.

    Клиент получает push-напоминание о заезде.

    Клиент может отменить запись.

    При отмене заезда центром клиент видит статус "Отменен" и получает уведомление.

    После завершения заезда клиент может поставить оценку маршалу.

    Клиент видит свои предстоящие и прошедшие заезды в профиле.
